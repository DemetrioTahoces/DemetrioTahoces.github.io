Diseño orientado al cambio

SOLID: principios de diseño para software que cambia

Una explicación práctica de SRP, OCP, LSP, ISP y DIP como herramientas para reducir acoplamiento, proteger contratos y evitar que cada cambio atraviese todo el sistema.

La escena típica

Cuando hablamos de código, la deuda técnica empieza igual en muchas ocasiones: una clase pequeña dentro de un servicio backend. Hace una cosa, se entiende rápido y nadie se queja. Dos meses después calcula precios, guarda datos, manda emails, publica eventos y tiene un if nuevo por cada caso raro que aparece. Aún funciona, pero ya da pereza tocarla.

Ahí es donde SOLID empieza a tener sentido. No como una religión de interfaces, sino como una forma de preguntar: si mañana cambia esta regla, ¿cuánto código arrastra detrás?

El problema real: cambiar sin romper medio sistema

El software no se desgasta como una pieza mecánica. Se vuelve difícil cuando cada cambio pequeño obliga a revisar demasiadas zonas. Suele pasar por tres motivos bastante mundanos:

  • responsabilidades mezcladas;
  • contratos poco claros;
  • dependencias hacia detalles volátiles.

Los principios vienen del diseño orientado a objetos, pero la idea aplica igual a servicios, módulos, casos de uso o paquetes. La unidad concreta importa menos que el olor: si para tocar una regla de negocio tienes que entender infraestructura, formato de salida y mensajería, algo se ha mezclado.

S: Single Responsibility Principle

La versión de pegatina dice: "una clase debe hacer una sola cosa". Está bien para empezar, pero se queda corta. La pregunta útil es: ¿quién pide que esto cambie?

Una clase InvoiceService que calcula impuestos, guarda facturas, genera PDF y publica eventos mezcla motivos de cambio distintos. Fiscalidad, almacenamiento, documento e integración no cambian por el mismo actor, aunque todo hable de facturas.

Esto no significa una clase por verbo. Si dos decisiones cambian siempre juntas, separarlas solo añade ceremonia. SRP pide cohesión alrededor de una razón real de cambio.

Señales rápidas

  • cambios pequeños obligan a tocar una clase enorme;
  • hay métodos usados por actores distintos;
  • los tests necesitan mocks de persistencia, formato, red y dominio a la vez;
  • el nombre contiene Manager, Processor o Helper y no explica una responsabilidad clara.

O: Open/Closed Principle

OCP suele sonar a promesa imposible: extender sin modificar. En la práctica significa algo más razonable: si una variación aparece cada dos semanas, dale un sitio claro donde entrar.

Piensa en descuentos. Primero hay un descuento VIP. Luego empleados. Luego Black Friday. Luego una campaña para partners. Si cada regla nueva acaba en el mismo método con otro if, ese método se convierte en una centralita telefónica.

Una política de descuento, un handler o una función registrada pueden dejar estable la mecánica y mover la variación a piezas pequeñas. No hace falta herencia para cumplir OCP.

Cuándo no merece la pena

  • si solo existe una variante y no hay señal real de cambio;
  • si la abstracción se diseña antes de entender el dominio;
  • si el punto de extensión oculta una regla que debería ser explícita.

OCP no significa "no modificar nunca". Significa no tocar código estable cada vez que aparece una variación previsible.

L: Liskov Substitution Principle

LSP no va de que el compilador acepte una subclase. Va de no sorprender al código que la usa.

Si una función recibe un PaymentMethod, debería poder trabajar con tarjeta, transferencia o wallet sin preguntar por el tipo concreto. Si una implementación devuelve null donde el contrato promete una lista vacía, o lanza una excepción que nadie esperaba, el contrato está roto aunque todo compile.

El ejemplo del pingüino y el método fly() sigue siendo útil por eso: el problema no es biológico, es contractual. Si no todos los pájaros vuelan, el tipo base no debe prometer vuelo.

Checklist

  • no fortalecer precondiciones en el subtipo;
  • no debilitar postcondiciones;
  • no romper invariantes del tipo base;
  • no exigir al cliente if concrete_type == ...;
  • no usar herencia para reutilizar código si no hay sustitución real.

I: Interface Segregation Principle

ISP es el principio del menú corto. Si solo quieres un café, no deberían obligarte a cargar con toda la carta del restaurante.

En código pasa con interfaces que mezclan creación, borrado, exportación, email y scoring. Un caso de uso que solo crea usuarios acaba acoplado a operaciones que no necesita. Eso contamina tests, permisos e implementaciones.

La separación buena no es una interfaz por método. Es una interfaz por cliente o caso de uso. Si dos operaciones se consumen siempre juntas y forman un contrato estable, pueden vivir juntas sin problema.

D: Dependency Inversion Principle

DIP es fácil de confundir con "usar inyección de dependencias". No es eso. La inyección es una técnica; la inversión es decidir quién define el contrato.

Las reglas importantes del sistema no deberían importar directamente el cliente HTTP, el ORM, Kafka, Redis o el proveedor cloud. El caso de uso define lo que necesita; la infraestructura se adapta.

Un RegisterUserUseCase debería depender de algo como UserRepository, no de PostgresUserRepository. El negocio no debería saber si hoy guardas en PostgreSQL y mañana en otro sitio.

Esto conecta directamente con arquitectura hexagonal:

  • el centro define puertos;
  • los adaptadores implementan detalles;
  • la composición decide qué implementación concreta se enchufa.

La idea encaja muy bien con arquitectura hexagonal: el centro define puertos y los adaptadores técnicos se enchufan fuera.

Cómo se refuerzan entre sí

Los principios no viven aislados:

  • SRP detecta ejes de cambio.
  • OCP convierte variaciones estables en puntos de extensión.
  • LSP protege que esas extensiones sean sustituibles.
  • ISP mantiene pequeños los contratos que cruzan límites.
  • DIP hace que las políticas importantes no dependan de detalles técnicos.

La gracia está en usarlos juntos, no en recitarlos por separado. SOLID empuja a que el cambio entre por zonas previstas y no atraviese todo el sistema como una gotera.

Errores habituales

Confundir SOLID con más capas

Más capas no implican mejor diseño. Si una interfaz solo tiene una implementación, no hay previsión de variación y solo añade navegación, quizá no aporta nada. La abstracción prematura también es deuda.

Usar herencia donde basta composición

Muchos problemas que se intentan resolver con jerarquías se expresan mejor con composición. LSP es una advertencia: si una subclase necesita desactivar o contradecir parte del padre, probablemente no hay relación de subtipo.

Aplicar OCP a todo

No todo merece un sistema de plugins. Conviene cerrar aquello que ya es estable y abrir aquello que cambia con frecuencia. El diseño debe seguir evidencia, no ansiedad.

Diseñar interfaces desde la implementación

Una interfaz que copia todos los métodos de una clase concreta no invierte nada. Solo duplica el detalle. Las buenas abstracciones nacen del consumidor: "qué necesita este caso de uso para cumplir su responsabilidad".

SOLID en código moderno

Aunque SOLID nació en el contexto de orientación a objetos, su valor no depende de escribir Java o C#. En TypeScript, Python, Go o arquitecturas funcionales siguen existiendo los mismos problemas: demasiadas razones de cambio, contratos excesivos, implementaciones que sorprenden al cliente y políticas de negocio atadas a detalles técnicos.

Un estudio reciente sobre código de Machine Learning encontró evidencia de que aplicar SOLID mejoraba la comprensión del código por parte de participantes del estudio. No lo convierte en dogma, pero sí refuerza una intuición sencilla: separar responsabilidades y contratos ayuda cuando otro tiene que leer tu código un martes por la tarde.

Regla pragmática

Antes de aplicar un principio, conviene preguntar:

  1. ¿Qué cambio concreto estoy intentando facilitar?
  2. ¿Qué parte del sistema quiero proteger?
  3. ¿La abstracción reduce acoplamiento real o solo desplaza complejidad?
  4. ¿El contrato es estable desde el punto de vista del consumidor?
  5. ¿Los tests quedan más simples o más ceremoniosos?

Si no puedes responder, probablemente estás aplicando SOLID como estética, no como ingeniería.

Conclusión

SOLID no garantiza buen diseño. Ayuda a razonar sobre cambio, acoplamiento y contratos. Su uso maduro consiste en aplicarlo donde el sistema muestra presión real: módulos que cambian por motivos distintos, reglas que necesitan nuevas variantes, jerarquías con contratos frágiles, interfaces demasiado amplias y casos de uso acoplados a infraestructura.

La mejor versión de SOLID no produce más código por principio. Produce código donde las decisiones importantes están protegidas, las variaciones tienen un lugar natural y los detalles reemplazables no gobiernan la arquitectura.
